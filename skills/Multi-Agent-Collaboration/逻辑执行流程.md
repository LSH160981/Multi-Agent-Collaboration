# Multi-Agent-Collaboration 逻辑执行流程（中文）

## 总体流程

### 第 1 步：用户下达任务

用户可以通过三种方式进入本 skill：

1. 直接自然语言提出复杂任务
2. 使用 `/mac XXX任务`
3. 明确说“使用 Multi-Agent-Collaboration skill 完成 XXX任务”

### 第 2 步：主Agent理解任务

主Agent 先做四件事：

1. 判断任务是不是复杂任务
2. 判断是否缺少关键信息
3. 判断任务需要哪些能力
4. 判断是否需要双组竞争

如果缺少关键信息，不进入执行阶段，先向用户追问。

### 第 3 步：激活核心控制层

主Agent 不会直接把任务丢给一堆 Agent，而是先激活：

- 主Agent 自身
- AgentPool（招聘/编组）
- 审核Agent（评分/驳回）
- 检查Agent（巡检/唤醒/恢复）

这是整个系统的控制层。

### 第 4 步：AgentPool 招聘人才

AgentPool 根据任务需要分析能力缺口：

- 现有 Agent 够不够用
- 哪些 Agent 可以复用
- 哪些 Agent 必须新建
- 哪些 Agent 需要被限制边界

接着创建两组团队：

- Group A
- Group B

每组至少有：

- 组长Agent
- 若干 specialist

每个 specialist 都应当是“看名字就知道干什么的” Agent，例如：

- A组-前端专家 Frontend-Agent
- A组-测试专家 Test-Agent
- B组-检索专家 Research-Agent
- B组-实现专家 Implementation-Agent

### 第 5 步：结构化派单

主Agent 把任务拆解成任务包，由组长继续向组内派发。

任务包必须至少包括：

- 任务目标
- 输入材料
- 约束条件
- 交付物要求
- 回复对象
- 当前状态

而且必须既写日志，也写任务队列，保证中断后能恢复。

### 第 6 步：双组并行竞争

A 组和 B 组并行工作。

这样设计的原因：

1. 避免单一路线思维固化
2. 提高结果质量上限
3. 允许审核Agent做横向对比
4. 给 Agent 权重系统提供真实反馈样本

### 第 7 步：检查Agent持续巡检

检查Agent 在执行阶段不能闲着。

它要不断检查：

1. Agent 最近有没有动作
2. 日志有没有新增
3. 队列状态有没有前进
4. 有没有卡死、超时、假装工作
5. 有没有该产生产物却没有产物

如果发现异常：

- 先发激活消息
- 再 retry
- 再重派单
- 再重建 Agent

目标不是“发现问题”，而是“让他们继续干活”。

### 第 8 步：审核Agent三维评分

当组长提交成果后，审核Agent从三个维度检查：

1. Reviewer：格式是否完整、交付是否齐全
2. Judge：内容质量是否达标，推理是否可靠
3. Metrics：能否验证，是否稳定，可执行性如何

评分后产生结果：

- 通过
- 驳回重做
- 通过但降权
- 通过并加权保留
- 整组淘汰并重建

### 第 9 步：主Agent最终汇总

主Agent 不能把内部所有过程都发给用户。

主Agent 必须先做消息治理：

- 去重
- 合并异步回执
- 删除无效中间状态
- 删除内部噪音
- 只保留最新有效结论

然后再向用户输出。

### 第 10 步：记忆、日志、git 留档

任务完成后，不是直接结束。

系统要继续做收尾：

1. 更新各 Agent 记忆
2. 更新任务队列状态
3. 保存关键产物
4. 把研究、流程、问题、改进建议写入文档
5. 用 git 记录这次变更
6. commit message 使用中文

## 失败恢复逻辑

### 1. Specialist 挂掉

读取其日志、队列、最近产物。

- 如果可继续，则恢复执行
- 如果不可继续，则创建替代 specialist，并继承必要上下文

### 2. 组长挂掉

从组内 specialist 的日志与队列恢复组内进度。
必要时重新指定组长。

### 3. 审核Agent 中断

保留已评分记录，恢复后从未完成的评分项继续。

### 4. 检查Agent 中断

优先恢复检查Agent，因为它负责整个系统的“活性”。

### 5. 主Agent 中断

主Agent 是唯一用户出口，所以恢复时必须先读取：

- 历史对用户发送过什么
- 当前任务状态
- 当前各组结果

防止恢复后给用户重复发相同意思的话。

## 为什么这是 OpenClaw 原生版

因为这里的 Agent 不是外部服务里的抽象 worker，而是 OpenClaw 的原生 session。

也就是说：

- 它们天然能调用 OpenClaw 工具
- 天然能独立保留上下文
- 天然能在 TUI / GUI / TG 等场景下复用
- skill 只是在这些原生能力之上加了一层协作纪律、组织结构和流程控制
