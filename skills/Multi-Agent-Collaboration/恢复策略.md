# 恢复策略（最终版）

恢复目标不是“重启一下看看”，而是：

- 找到最近有效状态
- 尽量保留已有产物
- 让任务继续推进
- 避免整套系统无脑从头重跑

---

## 1. 通用恢复顺序

任何 Agent 出问题，都按这个顺序处理：

1. 读 `queue/`
2. 读 `logs/`
3. 读最近产物 `artifacts/`
4. 找最后一次成功步骤
5. 判断是 retry / 重派 / 重建
6. 记录恢复动作与原因

---

## 2. stale 判断标准

如果满足以下任一情况，可视为 stale：

- 长时间无日志新增
- 任务队列状态长时间不推进
- 应有产物但始终无产物
- 多次只回执、不交付
- 连续卡在同一 blocker 且无恢复动作

检查Agent 发现 stale 后，动作优先级如下：

1. `agent_ping`
2. 重发任务上下文
3. retry
4. 重派给同角色替代 Agent
5. 重建 Agent

---

## 3. Specialist 挂掉怎么办

### 可恢复场景
- 上下文完整
- 队列明确
- 最近产物还在

动作：
- 原 Agent 继续
- 从最后完成步骤恢复

### 不可恢复场景
- 会话损坏
- 队列损坏
- 持续空转
- 连续多次失败

动作：
- 新建同角色 Agent
- 继承必要任务上下文
- 不盲目继承全部噪音历史

---

## 4. Lead 挂掉怎么办

组长出问题时：

1. 从组内 specialist 的 `queue/` 和 `logs/` 反推当前进度
2. 临时指定替代组长
3. 汇总未交付项
4. 继续组织本组执行

原则：
- 不因为组长挂掉就让整组清零

---

## 5. Group 整体失败怎么办

当整组都低效、失真、乱做时：

1. 审核Agent 记录失败原因
2. 检查Agent 记录恢复分析
3. AgentPool 解散该组
4. 重新招聘新组
5. 保留失败案例供后续学习与权重调整

---

## 6. Review 中断怎么办

审核流程中断时：

- 已评分条目保留
- 从未评分项继续
- 已做出的有效裁决不得重复评分造成漂移
- 评分卡应写入结构化文件

---

## 7. Inspection 中断怎么办

检查Agent 是系统活性中枢，优先恢复。

恢复动作：
1. 重载其最近巡检快照
2. 补扫各 Agent 最新状态
3. 重建待恢复列表
4. 继续巡检

---

## 8. 主Agent 中断怎么办

主Agent 恢复时必须先检查：

- 已给用户发过什么
- 当前任务最新状态
- 哪组已经通过审核
- 用户是否已经收到阶段性结论

目的：
- 避免重复回复
- 避免把旧结论再发一次
- 避免把内部草稿发给用户

---

## 9. 恢复后必须做什么

每次恢复后都要留下记录：

- 出了什么问题
- 造成了什么后果
- 采取了什么恢复动作
- 为什么这么恢复
- 以后如何预防
- 伪代码 / 流程 / 代码要不要更新

这一步会直接喂给自学习系统，用来让 skill 持续进化。
