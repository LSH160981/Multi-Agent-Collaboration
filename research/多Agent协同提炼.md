# 多 Agent 协同提炼

## 一、共识

从 OpenCrew、OpenMOSS、ClawHub skill 与相关文章里，可以提炼出几个稳定规律：

1. **单 Agent 迟早会被上下文拖垮**
2. **多 Agent 不能只靠“大家互相聊聊”**，必须有状态机、协议、角色边界
3. **异步通信比同步阻塞更稳**
4. **审核与巡检不是附属角色，而是主系统的一部分**
5. **任务要结构化、交接要结构化、收尾要结构化**
6. **用户看到的必须是去重后的最终结论，不是内部噪音**

## 二、优秀设计点

### 1. OpenCrew 的优点

- A2A 协议写得很细
- 可见性与审计链路设计清楚
- 知识沉淀分层
- 自主等级清晰

### 2. OpenMOSS 的优点

- 规划/执行/审查/巡查 四职能闭环
- 自组织、自动恢复、巡逻
- 评分和淘汰思维

### 3. 文章实践的优点

- 强调先做握手测试
- 强调 sessionKey 比 label 更稳定
- 强调复杂任务要隔离 session

## 三、我们自己的关键设计

### 设计原则

- **Agent = OpenClaw session**
- **主 Agent = 唯一用户出口**
- **先理解需求，再建人干活**
- **默认不预置一堆子 Agent，用到才招聘**
- **同题双组竞争，审核择优**
- **检查 Agent 持续巡检，强制唤醒不工作的 Agent**
- **所有 Agent 有独立记忆/日志/任务队列/能力文档**
- **所有用户可见消息必须先去重、去噪、合并异步回执**

## 四、伪代码应写什么

必须单独写中文文件，至少包括：

1. 任务进入后的总流程
2. AgentPool 招聘逻辑
3. 双组竞争逻辑
4. Agent 间 JSON 通信协议
5. 审核评分逻辑
6. 巡检与恢复逻辑
7. 自动自学习逻辑

## 五、代码逻辑应该长什么样

最稳的路线不是一上来做成复杂服务，而是：

1. **skill 文档先定义纪律**
2. **任务队列文件先落地**
3. **A2A JSON 消息协议先落地**
4. **再逐步补自动化工具与脚本**

这也是本仓库第一版采取的策略。
